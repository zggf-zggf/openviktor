## Unknown 1: The Skill Routing Algorithm

### What We Now Know (HIGH CONFIDENCE)

**The routing is LLM-native, not a separate system.** Viktor's skill routing is elegantly simple — it's the LLM itself doing the routing, not an embedding search or keyword matcher.

**Evidence chain:**

1. **Blog post confirmation** ([What Breaks When Your Agent Has 100,000 Tools](https://getviktor.com/blog/what-breaks-when-your-agent-has-100000-tools)): Viktor went through three iterations:
   - v1: Dump all tool schemas into context (abandoned — too wasteful)
   - v2: Search-based discovery (abandoned — "agents don't know what they don't know")
   - **v3 (current): One-line summaries with lazy loading** — ~18 core skills + one per integration = ~68 lines in the system prompt

2. **SKILL.md format spec** (`skills/skill_creation/SKILL.md`): Each skill has YAML frontmatter with `name` and `description` (max 1024 chars). The description format is: `[What it does]. Use when [trigger conditions].` Negative triggers are supported: `Do NOT use for CSV exports (use data_export skill instead).`

3. **Workspace tree** (`sdk/utils/workspace_tree.py`): Skills directory shows `names_only` handling — only skill folder names are listed, not internal files. This means the system prompt contains just skill names + descriptions.

4. **Loading mechanism**: When the LLM decides it needs a skill, it uses `file_read` to load the full SKILL.md body — instructions, code examples, gotchas, function signatures. This is confirmed by agent run traces showing `file_read` on SKILL.md files before tool calls.

**How routing actually works:**
- System prompt contains ~68 one-liner skill descriptions
- LLM reads the user request and decides which skill(s) are relevant
- LLM issues `file_read("/work/skills/{name}/SKILL.md")` to load full instructions
- If detailed docs needed: `file_read("/work/skills/{name}/references/{doc}.md")`
- No embedding search, no keyword matching, no confidence thresholds — it's pure LLM judgment

**When multiple skills match:** The LLM decides. The `description` field's "Use when" and "Do NOT use" clauses guide the decision. No priority system or confidence threshold exists because the LLM handles disambiguation natively.

**Managed vs custom skills:**
- Managed skills come from `backend/coworker/skills/**` and sync via `.sdk_managed_skills_manifest_v1.json`
- Custom content below `END OF AUTOGENERATED CONTENT` marker is preserved across syncs
- New integration skills are created by an agent that explores the API and writes the SKILL.md

### What Remains Unknown (LOW)

- **Exact system prompt assembly**: We know skill descriptions go into the system prompt but don't have the template that assembles them
- **Ordering**: Whether skills are alphabetical, by recency, or by some relevance ranking in the prompt
- **Token budget management**: How many skills can fit before the system prompt hits limits (68 lines ≈ ~200 tokens, trivial)

### Confidence: 90% — The mechanism is clear. Only prompt assembly details are missing.

## Unknown 2: The Thread Orchestrator's Internal State Machine

### What We Now Know (HIGH CONFIDENCE)

**The thread system is simpler than expected — it's HTTP-based spawning, not an event bus.**

**Complete API surface** (from `sdk/tools/default_tools.py` + `sdk/tools/thread_orchestration_tools.py`):

```python
# Spawn a new agent thread
create_thread(
    path: str,              # e.g., "/slack/Mateusz/1772007700_270919"
    title: str,             # Human-readable title
    initial_prompt: str,    # Full context for the new agent
    dependent_paths: list[str] | None  # Paths this thread depends on
) → {status, thread_id, path}

# Send message to existing thread
send_message_to_thread(
    content: str,
    thread_id: str,
    agent_runs_path: str | None,
    trigger_reply: bool = True  # Whether to wake the agent
) → {status, message_id}

# Block until other threads complete
wait_for_paths(
    paths: list[str],
    timeout_minutes: int = 30
) → {waited_seconds, paths_waited_for, timed_out}

# Query thread/cron state
list_running_paths() → {running_paths: list[str]}
get_path_info(path: str) → PathInfo(path_type, cron, thread)
```

**Thread lifecycle from agent run traces:**

1. **Trigger event** → Platform creates a new agent run with a system message containing:
   - `## Your Thread Info` with path, trigger type, task description
   - `## Currently Active Threads` listing all running threads
   - Full task.json `description` inlined for cron-triggered runs
   - `thread_lock` message immediately after system message

2. **Agent executes** → Makes tool calls via the tool gateway, reads/writes files

3. **Thread completion** → Agent finishes, platform marks thread as done

4. **Dependencies**: `dependent_paths` creates ordering. `wait_for_paths` is the coordination primitive — it blocks with a 30-minute default timeout and returns whether it timed out.

**How `wait_for_paths` likely works:** Given the 600s tool timeout and the return value `{waited_seconds, paths_waited_for, timed_out}`, this is almost certainly **server-side polling or long-polling** — the tool gateway holds the request and checks path status periodically, returning when all paths complete or timeout is reached. NOT webhooks, NOT an event bus.

**Evidence from HB#2** (the only `create_thread` in 59 runs): Heartbeat spawned a crypto research thread with full context in `initial_prompt`. The spawned thread ran independently. No `wait_for_paths` was called — the heartbeat continued without blocking.

**Thread path hierarchy:**
- `/slack/{user_name}/{timestamp}` — user-initiated DM threads
- `/{cron_name}/threads/{ISO_datetime}` — cron-spawned threads
- `/integration_exploration/{name}` — onboarding exploration threads
- `/onboarding` — initial setup thread

### What Remains Unknown (MEDIUM)

- **Thread registry persistence**: Where does `list_running_paths` read from? Database? File system?
- **Concurrent execution model**: What happens when heartbeat and user thread run simultaneously? Agent runs appear sequential per the transcript timestamps, but multiple threads CAN be active (`## Currently Active Threads` lists multiple)
- **Retry/failure handling**: No retry logic visible. If a thread crashes, does the platform restart it?
- **`thread_lock` message**: Appears in every agent run transcript immediately after the system message. Likely a mutex preventing concurrent writes to the same path, but mechanism unknown.

### Confidence: 80% — API is fully mapped. Server-side orchestration details remain opaque.

**Confirmed architecture:**
- Each agent run = new Modal Firecracker microVM with Named Volume mounted at `/work/`

- Container has full Linux userland, unrestricted outbound internet (no firewall rules observed)

- Environment variables injected per-run: `THREAD_ID`, `TOOL_TOKEN`, etc.

- Each event (Slack DM, cron tick, thread message) triggers a fresh Modal function invocation
- The agent reads files from volume (like loading from disk into RAM), executes, writes back, terminates

**Concurrent execution:**
- Multiple threads CAN be active simultaneously (system messages list "Currently Active Threads")
- Each gets its own container but shares the `/work/` Named Volume
- No file-level locking exists — conflicts mitigated by: append-only writes (`open(path, "a")`), dated subdirectories, per-run UUID temp dirs (`/work/temp/{UUID}/`)
- Inter-thread coordination uses `wait_for_paths()` (HTTP poll), not filesystem primitives

| `/work/sdk/` | Managed — auto-regenerated on integration changes |
| `/work/slack/` | Platform-maintained — webhook-synced Slack mirror |
| `/work/agent_runs/` | Platform-managed read-only |
| `/work/temp/{UUID}/` | Per-run ephemeral |

**Resource limits:**
- 600s tool call timeout (from `client.py`), 120s default bash timeout
- No explicit CPU/memory limits visible, but Modal has configurable resource classes
- Agent runs observed lasting 2-15 minutes typically

### What Remains Unknown (LOW)

- **Exact Modal resource class**: CPU/memory allocation per container
- **Cold start time**: Modal Firecracker boots in ~100ms typically
- **Volume consistency for concurrent writes**: Modal Named Volumes have eventual consistency — mitigation is architectural (append-only, partitioned paths)

### Confidence: 95% — Modal confirmed definitively via volume ID in traceback. Architecture fully mapped.

## Unknown 4: The Tool Gateway Server

### What We Now Know (HIGH CONFIDENCE)

**The gateway is a thin HTTP proxy that routes tool calls to backend services.**

**Complete client implementation** (`sdk/internal/client.py`):

```python
class ToolClient:
    # Singleton pattern
    gateway_url = os.getenv("TOOL_GATEWAY_URL", f"{os.getenv('API_URL')}/v1/tools")
    token = os.getenv("TOOL_TOKEN")

    async def call(self, role: str, **kwargs) -> Any:
        async with httpx.AsyncClient(timeout=600.0) as client:
            response = await client.post(
                f"{self.gateway_url}/call",
                headers={"Authorization": f"Bearer {self.token}"},
                json={"role": role, "arguments": kwargs},
            )
            if response.status_code != 200:
                raise Exception(f"Gateway error: {response.status_code} - {response.text}")
            data = response.json()
            if data.get("error"):
                raise Exception(f"Tool error: {data['error']}")
            return data.get("result")
```

**Key findings:**
- **Single endpoint**: `POST {gateway_url}/call` — ALL 166 tools route through this one URL
- **Routing**: The `role` field is the tool name (e.g., `"coworker_send_slack_message"`, `"mcp_posthog_query"`, `"pd_google_sheets_get_spreadsheet"`)
- **No client-side retry**: Zero retry logic. One shot, 600s timeout
- **No client-side rate limiting**: The SDK fires requests as fast as the agent wants
- **Bearer token auth**: `TOOL_TOKEN` environment variable, per-run

**Three integration types — confirmed from 3,142-entry integration catalog:**

| Prefix | Type | Count | Backend |
|--------|------|-------|---------|
| `coworker_*` / no prefix | Native | 28 | Viktor's own API services |
| `mcp_*` | MCP Protocol | 28 native | MCP server proxies (PostHog, Linear, Notion, GitHub, Salesforce, Stripe, Jira, etc.) |
| `mcp_pd_*` | Pipedream | 3,114 | Pipedream proxy (99.1% of integrations) |

Pipedream auth breakdown: 2,452 `keys` (78.7%), 632 `oauth` (20.3%), 30 `none` (1.0%).

**Role encoding rules** (from 2,530 observed tool calls):
- Native: `bash`, `file_read`, `coworker_send_slack_message`
- Linear: `mcp_linear_{snake_case}`
- Notion: `mcp_notion-{kebab-case}` (hyphen separator)
- PostHog: `mcp_posthog_{kebab-case}`
- Pipedream actions: `mcp_pd_{slug}_{pd_action_key_in_snake}`
- Pipedream proxy: `mcp_pd_{slug}_proxy_{get|post|put|patch|delete}`

**The gateway:**
1. Receives `{role, arguments}`
2. Routes by `role` prefix to the appropriate backend
3. For native tools: calls Viktor's internal APIs directly
4. For MCP tools: proxies to the appropriate MCP server with OAuth tokens
5. For Pipedream tools: proxies to Pipedream Lambda runtime (via `api.pipedream.com/v1/connect/{project_id}/components/configure`, project ID: `proj_gYsdLWq`)
6. Handles OAuth token refresh server-side (agents never see tokens)
7. Returns `{result}` or `{error}`

**Draft/approval state machine** (reconstructed from 33 drafts, 21 submissions, 12 approvals):

```
Agent calls mutating tool
    ↓
Gateway creates draft → {status: READY, draft_id: "mXJP..."}
    ↓
Agent calls submit_draft(draft_id, approval_code=None)
    → "This action requires approval"
    ↓
Agent sends Slack permission_request (Approve/Reject buttons)
    ↓
User clicks Approve → {approval_code: "approval_08fcaad7dff3bfdc"}
    ↓
Agent calls submit_draft(draft_id, approval_code="approval_08fcaad7dff3bfdc")
    → "MCP tool draft submission started" → {status: "FINISHED"}
```

One approval code can authorize multiple drafts simultaneously.

**Five error response formats** (27 errors / 10,584 messages = 0.26% rate):
1. Gateway HTTP error (non-200)
2. MCP JSON-RPC validation (`MCP error -32602`)
3. MCP upstream/network error (500, SSE failures)
4. Pipedream proxy passthrough (status codes in response body, `x-pd-proxy-error` header)
5. Pipedream configure endpoint error (404 from `api.pipedream.com`)

**Additional gateway-backed tools discovered:**
- `ai_structured_output` — routes to Gemini Flash for structured extraction (3 tiers: `fast`/`balanced`/`smart`)
- `quick_ai_search` — Google search + top-3 result summary
- `coworker_text2im` — image generation with cost tracking
- `create_custom_api_integration` — dynamic API integration creation
- `resolve_library_id` / `query_library_docs` — Context7 documentation lookup

**Bash tool**: 120s default timeout (not 600s), with message: *"Command timed out after 120 seconds."* The 600s ceiling applies to all other tools.

**Zero rate limiting observed**: 0 HTTP 429 responses in 10,584 messages across 76 runs.

### What Remains Unknown (LOW)

- **Server-side rate limiting**: None observed, but may exist at higher volumes
- **Credit deduction granularity**: Per-gateway-call or per-agent-run?
- **TOOL_GATEWAY_URL hostname**: Consumed from env but never logged (likely `api.getviktor.com`)

### Confidence: 95% — Client fully reverse-engineered. 2,530 tool calls analyzed. 5 error formats mapped.

## Unknown 6: The Slack Sync Pipeline

### What We Now Know (HIGH CONFIDENCE — MOSTLY RESOLVED)

**Hybrid pipeline: initial backfill via API + real-time webhook updates to flat files.**

**Complete file format** (from `sdk/utils/slack_reader.py`):

```
# Message format:
[1772007700.270919] @Mateusz Jacniacki: Pull our PostHog data and give me a summary...

# Metadata tags (appended to message):
[thread:1772007700.270919]   # Thread parent marker
[deleted:true]               # Deletion marker

# Regex: r"^\[([0-9.]+)\] @([^:]+): (.*)$"
```

**File structure:**
```
$SLACK_ROOT/                           # /work/slack/
├── {channel_name}/
│   ├── {YYYY-MM}.log                  # Monthly channel messages
│   └── threads/
│       └── {thread_ts}.log            # Per-thread messages
├── {person_name}/
│   ├── {YYYY-MM}.log                  # Monthly DM messages
│   └── threads/
│       └── {thread_ts}.log            # Per-thread DMs
├── all_your_sent_slack_messages.log   # Viktor's outbound log
└── at_mentioned_by_users.log          # @Viktor mentions
```

**Backfill mechanism** (`coworker_slack_history` tool):
- Fetches history for specified `channel_ids` with `range` and `end_date`
- Writes to the flat file structure
- Docstring: *"Files are automatically kept up-to-date via webhooks"*

**Real-time sync** (from `SLACK_ROOT` env var confirmed in agent runs):
- `$SLACK_ROOT` is set to `/work/slack`
- Agent runs show the variable is available and populated with current messages
- The `coworker_slack_history` comment about webhooks confirms real-time push to flat files

**Reader API** (`sdk/utils/slack_reader.py`):
```python
get_new_slack_messages(
    since: str,              # ISO timestamp
    channel_names: list,     # Filter channels
    include_threads: bool,   # Include thread messages
    max_messages: int         # Limit
) → dict  # Grouped by channel and thread
```

**Who writes the files:** The Viktor **platform** (not the agent) maintains the Slack sync. Evidence:
- Agent runs never show writes to `$SLACK_ROOT`
- `coworker_slack_history` is a tool call (goes through gateway), suggesting server-side file writing
- The `.log` files are pre-populated when agents start

**Freshness:** Near-real-time for messages in joined channels. The agent checks for new messages since its last heartbeat timestamp using `get_last_heartbeat_time()`.

**Global log confirms webhook events** (`logs/YYYY-MM-DD/global.log` has two distinct entry formats):
- **Platform events**: `[path] [event_type]` — includes `[webhook]`, `[slack_received]`, `[slack_sent]` markers
- **Agent entries**: `[ISO_TIMESTAMP] message` — free text from `log_action()`

The `[slack_received]` events confirm the platform writes to `.log` files in real-time as Slack Events API webhooks arrive. The `coworker_slack_history` backfill tool was never invoked in 15 days of captured runs — suggesting the platform auto-backfills on channel join.

### What Remains Unknown (LOW)

- **Exact webhook endpoint**: Slack Events API subscription URL on Viktor's backend
- **Latency**: Likely seconds (platform events appear in global.log near-instantly)
- **Edit handling**: `[deleted:true]` visible; edit representation unknown

### Confidence: 92% — Format, structure, pipeline shape, and webhook event types all confirmed.

## Unknown 8: Proactive Engagement Tuning Parameters

### What We Now Know (HIGH CONFIDENCE — FULLY RESOLVED)

**The behavioral rules are HARD-CODED in the system prompt, not learned from scratch.**

**Critical finding:** The heartbeat `task.json` description IS the behavioral configuration. It's a static prompt written at installation time, not a dynamic system.

**The task.json prompt says "be proactive":**
> "Your goal is to be VISIBLY helpful, not invisible. A heartbeat where you do nothing is often a missed opportunity."
> "Do At Least One Per Heartbeat"

**But LEARNINGS.md evolved to override it:**
> "This team communicates through code, not Slack. Viktor should be a tool they reach for, not the colleague that keeps knocking."

**This is NOT a platform-level parameter — it's emergent LLM behavior.** The agent read the environment signals and wrote new rules into LEARNINGS.md that contradicted the system prompt. The system prompt never changed.

### Tuning Parameters (All from task.json + LEARNINGS.md)

**Hard-coded in system prompt (task.json):**
- Heartbeat frequency: `1 8,11,14,17 * * *` (4x/day)
- Workflow discovery frequency: `1 9 * * 2,5` (Tue/Fri 09:01)
- "At Least One" proactive action per heartbeat
- 2+ hours unanswered question trigger
- Humor frequency: "maybe 1 in 3 heartbeats"
- DM for personal, channel for team-wide
- Spawn threads for deep work

**Emergent (learned by the agent, stored in LEARNINGS.md):**
- 5+ unread Viktor messages → stop outreach
- 7-day silence → reactive-only doctrine
- 8+ days zero engagement → STOP all proactive outreach
- 48hr abandon timer for tracked tasks
- Zero reactions + independent completion = negative signal
- Team's own product decisions as meta-signal (HUM-194 "agents should back off")
- Weekly digest as sole proactive touch after shutdown

**For reimplementation — the starting configuration:**

The system prompt (task.json) IS the starting configuration. It intentionally starts proactive ("be VISIBLY helpful") and relies on the agent discovering team preferences through LEARNINGS.md. This is by design:

1. **Start eager** → Cast a wide net, find what works
2. **Track signals** → Reactions, replies, independent completions
3. **Self-correct** → Write new rules into LEARNINGS.md
4. **Converge** → Reach steady state matching team's actual preferences

**The engagement decay timeline IS universal** — it's driven by the same signal-checking code (`check_engagement.py`) and the same rules in LEARNINGS.md. But the SPEED of decay is team-specific (this team was unusually unresponsive).

**The 5+ unread threshold SHOULD be parameterized** for reimplementation — it was discovered through Viktor's trial-and-error with this specific team. A more responsive team might tolerate 10+ unread before backing off.

**Academic validation** (from CHI 2025, NIH micro-randomized trials):
- Proactive AI increases task completion by 12-18% but reduces user preference for future proactive help when frequency is too high
- Notification engagement decays at 0.993/day (95% CI: 0.975-1.012) — after ~100 days, effect approaches zero
- Viktor's emergent behavior (high early → rapid pivot to low) matches the theoretically optimal pattern

**Competitor comparison:**
- **Glean Assistant**: Structures proactivity around 3 specific daily/weekly touchpoints (morning summary, plan-my-day, weekly snippets) — bounded by calendar, not open-ended
- **Moveworks Ambient**: Couples proactivity to workflow events only (approval pending, compliance deadline) — zero unsolicited insights
- **Standard Slack bots**: No engagement monitoring, same frequency forever. Viktor is architecturally superior — it reads its own output's reception.
